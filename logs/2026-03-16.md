### 💡 学びのポイント
**新規習得：**
- **`static` メソッド / フィールド：** インスタンス（実体）ごとではなく、クラス（設計図・工場）自体に紐づく機能やデータ。アプリケーションのユーティリティ関数（例: `Math.random()` のような便利ツール）として使われる。
- **VS CodeターミナルでのJS実行：** ブラウザを開かずとも、ターミナルから `node ファイル名.js` を実行することで、最速でロジックの動作確認ができる手法。

**理解深化：**
- **公式ドキュメントとの付き合い方：** すべてを「教科書」として熟読するのではなく、概要だけ掴んでコードを書き、詰まった時だけ「辞書」として参照する方が、開発スピードと定着率（ROI）が高い。
- **エラー出力の重要性：** 正常に動くコードを書くだけでなく、「あえて間違った書き方をしてエラー（`TypeError`）を出す」ことで、仕様の理屈を実体験として腹落ちさせることができる。

**トラブルシューティング：**
- **事象：** `static` メソッドをインスタンスから呼び出すとどうなるかの検証。
- **解決/結果：** `p1.checkWinner(p1, p2)` と記述して実行した結果、`TypeError: p1.checkWinner is not a function` が出力されることを確認。これにより「静的メソッドは実体からは呼べない」というMDNの説明を完全に理解した。

```javascript
// クラスとstaticメソッドの動作検証
class Player {
  constructor(name) {
    this.name = name;
  }

  // 通常のメソッド（インスタンスから呼ぶ）
  introduce() {
    console.log(`私は ${this.name} です。`);
  }

  // staticメソッド（クラスから直接呼ぶ）
  static checkWinner(playerA, playerB) {
    console.log(`${playerA.name} と ${playerB.name} の勝敗を判定します...`);
  }
}

const p1 = new Player("Alice");
const p2 = new Player("Bob");

// 【正常動作】
p1.introduce(); // "私は Alice です。"
Player.checkWinner(p1, p2); // "Alice と Bob の勝敗を判定します..."

// 【意図的なエラー検証】インスタンスからはstaticメソッドは呼べない
// p1.checkWinner(p1, p2); 
// -> TypeError: p1.checkWinner is not a function
```

### 💡 Key Learning Points
**Newly Acquired:**
- **`static` Methods/Fields:** Functions and data tied to the class (blueprint/factory) itself, rather than per instance (entity). Used as utility functions in applications (e.g., convenient tools like `Math.random()`).

- **Executing JS in VS Code Terminal:** This method allows for the fastest logic testing by running `node filename.js` from the terminal without opening a browser.

**Deepening Understanding:**
- **How ​​to Use Official Documentation:** Instead of reading everything as a "textbook," it's better to grasp the overview, write code, and refer to the documentation only when stuck, as this increases development speed and retention rate (ROI).
- **The Importance of Error Output:** Not only should you write code that works correctly, but by "intentionally writing incorrect code to produce errors (`TypeError`)," you can gain a real-world understanding of the logic behind the specifications.

**Troubleshooting:**
- **Issue:** Verification of what happens when a `static` method is called from an instance.
- **Solution/Result:** Confirmed that `TypeError: p1.checkWinner is not a function` is output when `p1.checkWinner(p1, p2)` is written and executed. This allowed me to fully understand the MDN explanation that "static methods cannot be called from instances."
