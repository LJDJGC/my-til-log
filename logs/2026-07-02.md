### 💡 学びのポイント
**新規習得：**
- **`?.`（オプショナルチェーン）**: `dayData?.byType[type]` は、`dayData` が null/undefined ならそこで評価を止めて undefined を返す。`if (!dayData)` のガードと組み合わせなくても安全にプロパティにアクセスできる。
- **`??`（null合体演算子）**: `value ?? 0` は「value が null または undefined のときだけ 0 を使う」という意味。`|| 0` との違いは、`||` は falsy 全般（0, "", false）を置き換えてしまうが、`??` は null/undefined だけを置き換える。

**理解深化：**
- **getLevel の設計思考**:
  1. 「引数で受け取った type の値を取り出す」→ `dayData.byType[type]`
  2. 「値がなければレベル0」→ `|| 0` or `?? 0` でガード
  3. 「0ならレベル0、閾値を超えるごとにレベルアップ」→ `if (value === 0) return 0;`
  4. 「最大値を超えたときのハンドリング」→ 最後に `return 4` で全範囲カバー
- **写経でも手を動かす価値**: ヒントありで書いても、自分で書くことで「読んだつもりでわかってなかった場所」が明確になる。今日は `getLevel` の引数に type があるのに使ってなかったことに自分で気づけなかった → 理解の穴を発見。

**トラブルシューティング：**
- **STEP 6の学習法について**: 完全な白紙からの再現にこだわりすぎず、「ヒントありで書く → フィードバックを受けて直す → 後日もう一度白紙でやる」のサイクルが効果的。1回で完璧を目指さなくていい。

### 💡 Key Learnings
**Newly Acquired:**
- **`?.` (Optional Chaining)**: `dayData?.byType[type]` stops evaluation and returns undefined if `dayData` is null/undefined. Allows safe property access without explicit `if (!dayData)` guard.
- **`??` (Nullish Coalescing)**: `value ?? 0` uses 0 only when value is null or undefined. Unlike `|| 0`, which replaces all falsy values (0, "", false), `??` only targets null/undefined.

**Deeper Understanding:**
- **getLevel Design Thinking**:
  1. Extract the specific type's value from dayData: `dayData.byType[type]`
  2. Guard against undefined: `|| 0` or `?? 0`
  3. Level 0 for zero activity, increment by threshold
  4. Catch-all return for values above max threshold
- **Even Copying Code Has Value**: Writing code with hints still reveals gaps in understanding. Today, not realizing the unused `type` parameter exposed a real understanding gap.

**Troubleshooting:**
- **STEP 6 Learning Method**: Don't stress about perfect recall from scratch. The cycle of "write with hints → receive feedback → fix → retry from scratch later" is more effective than one-shot perfection.
