### 💡 学びのポイント
**新規習得：**
- **`dateKey` 関数のフォーマットパターン**: Date型 → "YYYY-MM-DD" 文字列への変換。`String(d.getMonth() + 1).padStart(2, "0")` のイディオム。月は0始まり（0=1月）なので +1 が必要。
  ```tsx
  function dateKey(d: Date): string {
      return `${d.getFullYear()}-${String(d.getMonth() + 1).padStart(2, "0")}-${String(d.getDate()).padStart(2, "0")}`;
  }
  ```
- **非線形閾値の設計意図**: `getLevel` は ratio を 0.1 / 0.25 / 0.5 / 0.75 で区切る。なぜ等間隔（0.2刻み）にしないのか？→ 等間隔だと「ちょっとだけ勉強した日」もレベル1になり草が生えすぎる。最初の閾値を 0.25 に上げることで「ある程度やった日」だけが色づく。**UIの「見せたい情報量」を調整するための設計判断**。
- **`aggregateByDate` の2パス設計の真意**: 1パス目で全セッションを日付ごとに集約（total合計・byType内訳）。2パス目で各日の `dominantType` を決定。**なぜ1ループで同時にやらないのか？**→ 「集計」と「判定」は別の責務。1ループでやろうとすると「もし同点だったら？」「まだ集計中のデータで判定していいのか？」という判断が混ざる。責務を分けることでコードの正しさを担保しやすい。
- **`Map` を選ぶ理由（vs オブジェクト `{}`）**: `Map.has()` でキーの存在確認が明確。`.values()` で値だけのループができる。forEachのメソッドチェーンが書きやすい。値の追加・更新が `map.set(key, value)` で統一されている。
- **`useMemo` チェーンの依存関係**: 4つの useMemo が直列に繋がる構造：
  ```
  sessions → { dayMap, maxDayMinutes }  [sessions依存]
  weeks    → days[]                      [weeks依存]
  days[]   → weeksList[][]               [days依存]
  weeksList → monthLabels[]              [weeksList依存]
  ```
  **なぜ4つに分けるのか？** → 1つの巨大な useMemo に全部詰め込むと、sessionsが変わると全部再計算される。分けることで「変わったところだけ再計算」を実現。
  **なぜ useMemo が必要か？** → React はコンポーネントが再レンダリングされるたびに関数内のコードを全部再実行する。`useMemo` をつけると「依存配列の値が変わった時だけ再計算」になり、無駄な処理をスキップできる。

- **復習ブランチの設計**: 型定義（DayData, Props）と dateKey 関数だけ残し、実装をすべて削除。ヒントコメントとして「何を書くべきか」だけを記載。コードを「読める」から「書ける」にするためのプラクティス設計。

### 💡 Key Learnings
**Newly Acquired:**
- **Why Non-Linear Thresholds in getLevel**: The ratio breakpoints at 0.1 / 0.25 / 0.5 / 0.75 are deliberately uneven. Equal intervals (e.g., 0.2 each) would make even light-study days appear at level 1, creating visual noise. The steeper first threshold (0.25) ensures only meaningful activity gets colored. **A UI design decision, not a math one.**
- **Why 2-Pass aggregation**: Pass 1 collects total duration and per-type breakdown per date. Pass 2 determines dominantType. These are separate concerns — trying to do both in one loop introduces edge cases ("what if there's a tie?", "is the data complete yet for this date?"). Separating concerns makes correctness easier to verify.
- **Why Map over Object `{}`**: `Map.has()` for explicit key existence checks. `.values()` for iterating values directly. Consistent `.set()`/`.get()` API. Cleaner for dynamic key-value collections.
- **Why 4 separate useMemo hooks**: One monolithic useMemo would recompute everything whenever any dependency changes. Splitting creates granular dependency tracking — changing `sessions` doesn't re-run the date range calculation.
- **Why useMemo at all**: React re-runs the entire component function on every render. useMemo caches the result and only recomputes when dependencies change, preventing expensive recalculations.
- **Recall Branch Design Pattern**: Keep type definitions and structural helpers (dateKey), remove all implementation bodies. Add minimal hint comments indicating WHAT to write without revealing HOW. Transforms "I can read this code" into "I can write this code."

**Deeper Understanding:**
- **Transcription as Typo Magnifier**: Hand-typing reveals errors that the eye skips when reading:
  - `spawn` import from `child_process` (phantom import)
  - `bg-gren-*` → missing 'e' in "green"
  - `ring-offset--1` → double hyphen
  - `inLine-block` → `inline-block`

- **Learning to Ask "Why"**: Understanding *why* code is written a certain way (e.g., non-linear thresholds control visual density) is more valuable than memorizing *what* the code says. It enables customization and transfer to new contexts.
- **React Data Flow Epiphany**: The initial confusion ("why put variables in useMemo?") resolved by understanding React's render cycle — the component function re-runs entirely on every render. useMemo is not optimization; it's necessary caching.
