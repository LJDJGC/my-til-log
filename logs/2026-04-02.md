### 💡 学びのポイント
**新規習得：**
- **イミュータビリティ（不変性）：** 既存のデータ（配列やオブジェクト）を直接書き換える（破壊的変更）のではなく、コピーを作成して新しいデータとして置き換えるReactの基本原則。

**理解深化：**
- **なぜ直接書き換えてはいけないのか（2つの巨大なメリット）：**
  1. **履歴の保持（タイムトラベル）：** 元のデータを壊さないため、「一つ前の状態に戻す（Undo）」などの機能が圧倒的に実装しやすくなる。
  2. **再レンダリングの最適化：** Reactは「データが変わったか」を検知して画面を更新する。コピーを作って置き換える方式なら「以前のデータの入れ物と、新しいデータの入れ物が違うか」を見るだけで済むため、変更検知が非常に安価（高速）になり、パフォーマンスが向上する。
- **Vanilla JSとの比較：** JSの基礎学習時に `this.board[index] = marker` と直接書き換えていた処理を、Reactでは `slice()` でコピーしてから更新するという作法の違いを深く理解した。

**トラブルシューティング：**
- **事象:** 「朝4時に起きられない」という目標と現実のギャップによる自己嫌悪と焦り。
- **解決策:** 睡眠負債（夜更かし）が根本原因であると分析し、「4時という数字」への執着を捨て、「最低7時間の睡眠確保」と「手持ちの朝時間で最高のアウトプットを出す」というROIベースの思考へ切り替えた。

### 💡 Learning Points
**Newly Acquired:**
- **Immutability:** A fundamental principle of React that replaces existing data (arrays and objects) with new data by creating a copy, rather than directly overwriting it (destructive modification).

**Deepened Understanding:**
- **Why Direct Overwriting Shouldn't Be Used (Two Huge Benefits):**
1. **History Preservation (Time Travel):** Because the original data is not destroyed, features such as "Undo" become significantly easier to implement.
2. **Optimized Re-rendering:** React updates the screen by detecting whether "data has changed." Using a copy-and-replace method means that change detection is extremely inexpensive (fast) and performance improves because it only requires checking if "the container for the previous data and the container for the new data are different."
- **Comparison with Vanilla JS:** I gained a deeper understanding of the difference in conventions between directly rewriting `this.board[index] = marker` during basic JS learning and using `slice()` to copy and update in React.
