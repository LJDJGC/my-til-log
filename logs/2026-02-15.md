### 💡 学びのポイント
**新規習得：**
- **ファクトリー関数 (Factory Functions):** `new` キーワードを使わず、オブジェクトを戻り値として返す関数。クラス構文よりも柔軟で、`this` の挙動に悩まされにくい。
- **モジュールパターン (Module Pattern):** 即時関数 (IIFE) を使って、特定の処理をひとまとめにするパターン。アプリケーション内で一つしか存在しないもの（例：ゲームコントローラー）に適している。
- **クロージャによるカプセル化:** 関数内の変数は外部から直接アクセスできない性質（スコープ）を利用し、プライベート変数を作成する方法。

**理解深化：**
- **クラス vs ファクトリー:** クラス（コンストラクタ）はインスタンスを量産するのに向いているが、ファクトリー関数はよりシンプルにオブジェクト生成ができる。
- **IIFE (即時関数):** 定義と同時に実行される関数 `(() => { ... })();` の仕組みと、変数のスコープ汚染を防ぐメリット。

**トラブルシューティング：**
- **課題:** クロージャやスコープの概念が抽象的で、完全に腹落ちしていない感覚がある。
- **解決:** 理論だけで理解しようとせず、次の「Tic Tac Toe」プロジェクトで実際にコードを書きながら、変数の隠蔽（プライベート化）やパターンの使い分けを体感する方針に切り替えた。

### 💡 Learning Points
**New Learnings:**
- **Factory Functions:** Functions that return objects without using the `new` keyword. They are more flexible than class syntax and are less cumbersome with `this` behavior.
- **Module Pattern:** A pattern that uses an immediate implementation function (IIFE) to group specific processes together. Suitable for things that exist only once within an application (e.g., a game controller).
- **Encapsulation with Closures:** A method for creating private variables by taking advantage of the fact that variables within a function cannot be directly accessed from outside (scope).

Deepening Understanding:
- **Classes vs. Factories:** Classes (constructors) are suitable for mass-producing instances, but factory functions allow for simpler object creation.
- **IIFE (Immediate Function):** The mechanics of functions that are executed immediately upon definition, `(() => { ... })();`, and the benefits of preventing variable scope pollution.

**Troubleshooting:**
- **Challenge:** The concepts of closures and scope are abstract, and I don't feel like I fully understand them.
- **Solution:** Rather than trying to understand them theoretically alone, I decided to switch to a strategy of actually writing code in the next "Tic Tac Toe" project to experience variable hiding (privatization) and the different patterns used.

**理解深化：**
- **ディレクトリ構造の重要性:** ローカルのフォルダ構成がそのままGitHubの階層に反映されるため、プロジェクト作成時のディレクトリ作成（`mkdir`）の階層意識が重要であると実感した。

**トラブルシューティング：**
- **リポジトリの入れ子（マトリョーシカ）状態の解消:** 親ディレクトリで `git init` してしまった場合の対処法として、`rm -rf .git` で一度リセットし、正しい階層で初期化し直す手順を実践した。
- **Gitコマンドの構文ミス:** `git remote add origin [URL]` のように、正確なコマンドとリポジトリURL（`.git`で終わるもの）を指定する必要性を学んだ。
