### 💡 学びのポイント
**新規習得：**
- **JSX の3ブロック構成の意図**: GrassCalendar の return 文は「凡例（タイプ別の色説明）」「週グリッド（草の本体）」「サマリー（合計値表示）」の3ブロックに分かれている。これは「何を表示するか」を視覚的に整理するための設計。それぞれのブロックが独立しているので、例えば「サマリーを別コンポーネントに切り出す」といったリファクタリングが容易。

**理解深化：**
- **useMemo チェーンの実装理解**: sessions → dayMap → days → weeksList → monthLabels の4段チェーンを実際にコメントで説明しようとすることで、「なぜ4つに分けるのか」の理解が深まった。sessions が変われば dayMap だけ再計算、weeks が変われば days だけ再計算、と依存関係に沿った最小限の再計算が行われる仕組み。1つの巨大な useMemo に全部詰め込むと、どの値が変わっても全部再計算されて無駄が発生する。
- **dateKey フォーマットのイディオム**: `d.getFullYear() + "-" + (d.getMonth()+1).toString().padStart(2, "0") + "-" + d.getDate().toString().padStart(2, "0")` は TypeScript/JavaScript で日付を "YYYY-MM-DD" 文字列に変換する定石。月が0始まりなのがトラップで、+1 を忘れがち。`padStart(2, "0")` で1桁の月/日を2桁に揃える。

### 💡 Key Learnings
**Newly Acquired:**
- **JSX 3-Block Design Intent**: The return statement has three visual blocks: (1) legend showing type colors, (2) weekly activity grid (the "grass"), (3) summary totals. Each block is independent, making future refactoring straightforward — e.g., extracting the summary into its own component.

**Deeper Understanding:**
- **useMemo Chain in Practice**: Writing comments on the 4-stage chain (sessions → dayMap → days → weeksList → monthLabels) solidified the understanding of granular dependency tracking. Only the part that changes gets recomputed. A single monolithic useMemo would recompute everything on any change.
- **dateKey Format Idiom**: `getFullYear() + "-" + padStart(2, (getMonth()+1)) + "-" + padStart(2, getDate())` is the standard TypeScript pattern for "YYYY-MM-DD" formatting. Month is zero-indexed (trap: +1 required). `padStart(2, "0")` ensures single-digit months/days are zero-padded.
